# Implementation Plan ‚Äî ADR 0001: Node Groups & Services

Companion to `0001-node-groups-and-services.md`. Phased so each phase is independently shippable and reversible.

**Total estimate:** ~20‚Äì25 engineering hours, ideally across 3‚Äì4 PRs.

---

## Phase 1 ‚Äî Schema + types + sqlc (no behavior change) ‚Äî ~4h

**Goal:** the new tables exist, are queryable via generated code, and nothing else has moved.

1. **Migration** `pkg/db/migrations/0022_add_node_groups_and_services.up.sql` (and `.down.sql`)
   - `node_groups`, `services`, `service_backups`, `service_events` (exactly as ADR schema).
   - `ALTER TABLE nodes ADD COLUMN node_group_id INTEGER REFERENCES node_groups(id) ON DELETE CASCADE`.
   - New `node_type` enum rows: `FABRICX_ORDERER_ROUTER`, `..._BATCHER`, `..._CONSENTER`, `..._ASSEMBLER`, `FABRICX_COMMITTER_SIDECAR`, `..._COORDINATOR`, `..._VALIDATOR`, `..._VERIFIER`, `..._QUERY_SERVICE` (if enums are enforced via CHECK ‚Äî confirm against current `0020_*.up.sql`).
   - **Do NOT drop** `FABRICX_ORDERER_GROUP` / `FABRICX_COMMITTER` from the node type enum yet (needed for legacy rows until Phase 6).
   - Down migration drops in reverse.

2. **sqlc queries** in `pkg/db/queries.sql` (single file ‚Äî matches existing convention):
   - `NodeGroup`: Create, Get, List, ListByPlatform, UpdateStatus, Update, Delete, CountByStatus
   - `Service`: Create, Get, List, ListByNodeGroup, UpdateStatus, Update, Delete
   - `ServiceBackup`: Create, Get, ListByService, UpdateStatus, DeleteOlderThan
   - `ServiceEvent`: Create, ListByService
   - Extend node queries: `ListNodesByGroup(group_id)`, `UpdateNodeGroupID`
   - Regenerate: `sqlc generate`

3. **Go types**
   - `pkg/nodegroups/types/types.go` ‚Äî `NodeGroup`, `GroupStatus`, `GroupType` constants.
   - `pkg/services/types/types.go` ‚Äî `Service`, `ServiceType`, `ServiceStatus`, `BackupConfig`.
   - Extend `pkg/nodes/types/types.go` with the new child `NodeType` constants.

4. **Build verification:** `go build ./...` green. No handlers wired yet. No UI changes.

**Exit criteria:** migrations apply cleanly on a fresh DB and on the live `data/chainlaunch.db`. Rolling back and re-applying works. `sqlc generate` produces code. Nothing else is observably different.

## Phase 2 ‚Äî Backend orchestration: per-child lifecycle ‚Äî ~6h

**Goal:** the FabricX package stops pretending a group is one thing. Each child runs in its own goroutine/container lifecycle, owned by a real `nodes` row.

1. **Refactor `pkg/nodes/fabricx/orderergroup.go`**
   - Split the monolithic `Start()` / `Stop()` loops into per-role `StartRouter()`, `StartBatcher()`, etc.
   - Shared genesis / crypto / config generation stays on `OrdererGroup` (now renamed `OrdererGroupCoordinator` or kept as-is ‚Äî just scoped to orchestration, not runtime).
   - `Init()` now returns a `(groupConfig, []childDeploymentConfig)` pair.

2. **Refactor `pkg/nodes/fabricx/committer.go`** ‚Äî same pattern. Note: `startPostgres()` is pulled out entirely in Phase 2.5.

3. **New package `pkg/nodegroups/service/`** ‚Äî group-level coordinator
   - `CreateGroup(ctx, req)` ‚Äî writes the `node_groups` row, generates shared crypto, then writes one `nodes` row per child with correct `node_group_id` + role-specific config.
   - `StartGroup(ctx, id)` ‚Äî loads group + children, starts services first (Phase 2.5), then children in dependency order. Updates `node_groups.status` from aggregated child statuses.
   - `StopGroup`, `RestartGroup`, `DeleteGroup` ‚Äî symmetric.
   - Aggregated status derivation: `all RUNNING ‚Üí RUNNING`, `any RUNNING & any STOPPED ‚Üí DEGRADED`, etc.

4. **Wire child operations into existing node lifecycle**
   - `NodeService.StartNode` on a child already works (it's just another `nodes` row). The only extension: when the child's config is a `FabricXChildConfig`, dispatch to role-specific container spin-up in the fabricx package.
   - Per-child logs/events/metrics come for free from the existing node plumbing.

5. **HTTP handlers** ‚Äî new package `pkg/nodegroups/http/`
   - `GET /api/v1/node-groups`, `POST`, `GET /{id}`, `DELETE /{id}`, `POST /{id}/start|stop|restart`.
   - Register in `cmd/serve/serve.go` alongside `nodesHandler.RegisterRoutes(r)` (~line 612).
   - Swag annotations (`// @Summary`, etc.) matching existing handler style.

6. **OpenAPI regen + TS client regen**
   - `swag init -g cmd/serve/serve.go -o docs ...` (confirm exact command from project Makefile/scripts)
   - Regenerate `chaindeploy/web/src/api/client/` from the new OpenAPI.

**Exit criteria:** a new FabricX orderer group can be created via `POST /api/v1/node-groups`, shows 4 child `nodes` rows, each startable/stoppable individually, and the aggregated group status is correct. No frontend work yet ‚Äî verified by curl + DB inspection.

## Phase 2.5 ‚Äî PostgreSQL service with WAL-G ‚Äî ~6h

**Goal:** the committer's postgres becomes a first-class managed service with continuous archiving and PITR.

1. **New package `pkg/services/postgres/`**
   - `postgres.go` ‚Äî container lifecycle for `postgres:16-alpine` (template `postgresql.conf` with `archive_mode=on`, `archive_command=wal-g wal-push %p`, `wal_level=replica`).
   - `walg.go` ‚Äî sidecar container (`wal-g/wal-g:latest` or pinned) sharing the postgres data volume and `WALG_S3_PREFIX` env from service config.
   - `backup.go` ‚Äî `TriggerBaseBackup(serviceID)` runs `docker exec walg-sidecar wal-g backup-push /var/lib/postgresql/data`; creates a `service_backups` row, updates on success/fail, emits `service_events`.
   - `restore.go` ‚Äî stops postgres, wipes volume, runs `wal-g backup-fetch` + optional `recovery.conf` with `recovery_target_time`, restarts.

2. **Credential reuse**
   - `services.backup_target_id` ‚Üí join to `backup_targets`.
   - When spinning up the WAL-G sidecar, decrypt the target's `access_key_id` / `secret_key` / `endpoint` / `region` and pass as env.
   - No new credential storage; reuse `pkg/backups/service` encryptor.

3. **Scheduling**
   - Daily base backup: reuse the existing cron-like scheduler that drives `backup_schedules` if reasonable, OR spawn a per-service ticker inside `pkg/services/postgres/scheduler.go`. Decision in Phase 2.5 kickoff ‚Äî lean toward reusing existing infrastructure.
   - Retention enforcement: nightly sweep deletes WAL + base backups older than `backup_config.retention_days` from S3 (via `wal-g delete before FIND_FULL TIMESTAMP`) and from `service_backups`.

4. **HTTP handlers** ‚Äî `pkg/services/http/`
   - Full service CRUD + `/backups` subtree + `/restore`.
   - Guard: restore refuses if any child of the attached `node_group` is `RUNNING`.

5. **Committer group wiring**
   - `CreateGroup(type=FABRICX_COMMITTER)` auto-creates a `POSTGRES` service attached to the group.
   - `StartGroup` starts the postgres service, waits for `pg_isready`, then starts children.
   - The 5 committer children reference `postgresHost:port` from the service's `deployment_config`.

6. **Dev/local-dev parity**
   - `CHAINLAUNCH_FABRICX_LOCAL_DEV=true` behavior must apply to the postgres service too (host-gateway mapping for `externalIP`).

**Exit criteria:** creating a committer group provisions postgres with WAL archiving active. A manual `POST /services/{id}/backups` produces a base backup visible in S3. A destructive test (delete postgres volume ‚Üí restore) brings the committer back to a consistent state.

## Phase 3 ‚Äî Frontend ‚Äî ~4h

**Goal:** UI reflects the new model. Old monolithic detail view stays wired behind a flag until Phase 4.

1. **New pages**
   - `pages/node-groups/[id].tsx` ‚Äî group detail: header cards (type, MSP, party, status-aggregated), children list (table of child nodes with status + quick-actions), services list (postgres with backup history + actions).
   - `pages/node-groups/[id]/children/[nodeId].tsx` ‚Äî thin wrapper that reuses the existing `NodeDetailPage` for child-node rendering (logs, metrics, events per role).
   - `pages/services/[id].tsx` ‚Äî service detail: status, config, backup history table, restore dialog.

2. **New components**
   - `components/nodes/NodeGroupDetails.tsx` ‚Äî replaces the group-branch of the old FabricX detail view.
   - `components/services/PostgresServicePanel.tsx` ‚Äî postgres-specific rendering.
   - `components/services/BackupHistoryTable.tsx` ‚Äî reusable for any service.

3. **Routing** in `chaindeploy/web/src/App.tsx`
   - Add lazy-loaded `NodeGroupDetailPage`, `ServiceDetailPage`.
   - `Route path="/node-groups/:id"` and `/services/:id`.

4. **Existing node detail page (`pages/nodes/[id].tsx`)**
   - If `node.nodeGroupId` is set, render a "Part of group: [link]" breadcrumb above the detail.
   - Keep the FabricX* existing detail components around as dead code only for the deprecation window; remove in Phase 6.

5. **Navigation**
   - Add a "Node Groups" item under Platform in the sidebar; hide behind a feature flag or always-on if ready.

**Exit criteria:** a user can navigate Networks ‚Üí FabricX network ‚Üí click a committer group ‚Üí see children + postgres with backup history ‚Üí drill into any child for its own logs/metrics ‚Üí trigger a manual backup.

## Phase 4 ‚Äî Data migration script ‚Äî ~2h

**Goal:** the 8 live FabricX group rows (4 orderer groups + 4 committers, per current DB) split into 8 `node_groups` rows + 36 child `nodes` rows + 4 `services` rows, without downtime of the running containers.

1. **Script** `cmd/migrate-fabricx-groups/main.go`
   - `--dry-run` prints the plan: for each legacy node, which group row + children rows will be created.
   - `--db=path` (required). Takes a file-level backup first (`cp data/chainlaunch.db data/chainlaunch.db.pre-0001-migration`).
   - For each legacy `nodes` row where `node_type IN ('FABRICX_ORDERER_GROUP','FABRICX_COMMITTER')`:
     1. Parse its `node_config` and `deployment_config`.
     2. Insert a new `node_groups` row with shared fields.
     3. Insert N child `nodes` rows (one per role), each with role-specific port and `node_group_id` set. The child rows link to the already-running containers by matching `container_name` derived from the existing naming convention.
     4. For committers: also insert a `services` row for postgres, linking to a default `backup_targets` if one exists (otherwise leave `backup_target_id` NULL with a warning).
     5. Mark the legacy group row as `DEPRECATED` (new status or new column `legacy=true`) rather than deleting ‚Äî removes in Phase 6.
   - Transactional per group. On failure, rollback and log.

2. **Smoke test after migration**
   - `GET /api/v1/node-groups/{id}` returns the migrated group.
   - `GET /api/v1/nodes/{childID}` returns each child with `nodeGroupId` set.
   - Containers keep running; no restarts triggered by the migration itself.

**Exit criteria:** on the live dev DB, the 8 groups are migrated, children correspond to the actually-running containers, a manual group stop/start cycles the right containers.

## Open questions (resolve before Phase 2.5)

1. **Scheduling reuse** ‚Äî does the existing `backup_schedules` cron-runner work for service backups, or spawn a new ticker? Impact: ~1h either way.
2. **WAL-G version pinning** ‚Äî which WAL-G tag matches postgres 16 on the fabric-x path? Needs a quick image audit.
3. **Default backup target** ‚Äî for migrated committers, do we pick the first `backup_targets` row automatically, or require the operator to set one post-migration? I lean on the latter ‚Äî surfacing "backup not configured" as an orange-state service badge until the operator fixes it.
